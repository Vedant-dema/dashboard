## FEATURE-061: Shared Demo Customers Backend Sync (Vercel-visible)
**Date:** 2026-03-27
**Author/Agent:** Cursor AI
**Status:** Added

### What Was Added
A temporary shared demo backend sync flow was added for customers so data entered by one user becomes visible to another user on Vercel. This enables a realistic manager demo with shared state while keeping existing local mode as fallback.

### Where It Was Added
- `backend/main.py` — added shared demo customers endpoints (`GET/PUT /api/v1/demo/customers-db`), sqlite-backed key-value persistence, optional demo API key enforcement, and demo store initialization
- `frontend/src/store/kundenStore.ts` — added API-mode toggle and shared load/save functions (`isCustomersApiMode`, `loadSharedKundenDb`, `saveSharedKundenDb`)
- `frontend/src/pages/CustomersPage.tsx` — added startup sync from shared backend and optimistic persist with remote sync fallback
- `frontend/src/contexts/LanguageContext.tsx` — added `customersSyncFailed` i18n key in DE/EN/FR for shared-sync failure alert
- `docs/feature-logs/2026-03-27.md` — logged feature change entry
- `docs/FEATURE-LOG.md` — updated daily and master feature index

### What It Does (Technical)
1. Initializes a sqlite-backed demo store table on backend startup.
2. Exposes shared customers DB blob endpoints protected optionally by `x-demo-key`.
3. Enables frontend API mode via env flag `VITE_CUSTOMERS_SOURCE=api`.
4. On Customers page load, fetches shared state; if missing, seeds backend from local seeded data.
5. On each customer mutation, saves locally and then syncs to shared backend so other sessions can read the same updates.

### Data It Accepts / Emits
| Field | Type | Required | Description |
|----|---|----|----|
| `state` | object (`KundenDbState`) | Yes | Full customers DB blob shared across sessions |
| `x-demo-key` | string | No | Optional header required when `DEMO_API_KEY` is configured |
| `updated_at` | string (ISO datetime) | No | Backend write timestamp returned with shared state payload |
| `VITE_CUSTOMERS_SOURCE` | string | Yes (for API mode) | `api` enables shared backend sync; other values keep local mode |
| `VITE_API_BASE_URL` | string | Yes (for API mode) | Frontend base URL for shared demo backend |

### Database
- **Engine:** SQLite (backend, file path from `DEMO_CUSTOMERS_DB_PATH` or default `demo_shared.db`)
- **Tables Affected:** `demo_store`
- **Schema Changes:** Added key-value table with columns `key`, `value`, `updated_at`
- **Key Queries:** Upsert/read single key `customers_db`

### API Endpoints (if applicable)
| Method | Path | Auth | Request Body | Response |
|-----|---|---|----|----|
| GET | `/api/v1/demo/customers-db` | Optional `x-demo-key` | N/A | `{ state, updated_at }` |
| PUT | `/api/v1/demo/customers-db` | Optional `x-demo-key` | `{ state }` | `{ state, updated_at }` |

### State / Store (if applicable)
- **Store file:** `frontend/src/store/kundenStore.ts`
- **Actions/Selectors added:** `isCustomersApiMode`, `loadSharedKundenDb`, `saveSharedKundenDb`
- **Persisted:** Yes (localStorage + shared backend in API mode)

### i18n Keys Added
- `customersSyncFailed`

### Dependencies Added
None.

### Notes / Known Limitations
This is intentionally temporary demo architecture using whole-state blob sync (not fine-grained concurrency-safe domain APIs). Suitable for shared demos and can be replaced by normalized endpoint-based production services later.
